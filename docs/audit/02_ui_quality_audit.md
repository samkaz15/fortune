## 2. UI監査(Blueprintとの比較、画面ごとの重大度評価)

実機スクリーンショットとの比較は本環境では取得不可のため、**コードレベルでの構造的な一貫性検証**を行った。「崩れ」の断定ではなく「崩れのリスクが構造的にある箇所」として報告する。

### 評価サマリー(画面横断の観測項目)

| 観測項目 | 状態 | 重大度 |
|---|---|---|
| Bottom Navigation | 全26ページで`layout.tsx`から共通コンポーネント`BottomNav.tsx`を使用。LP(v4静的HTML)ともCSS値レベルで統一済み(2026-07-06対応) | 問題なし |
| Header | 全ページ共通。ログイン状態のsessionStorageキャッシュあり(体感速度対策済み) | 問題なし |
| Card(rounded-card) | `rounded-card`(20px)トークンが各ページで使われているが、**一部ページはインラインで`rounded-2xl`等別の角丸を直書き**(例: love/work/selfのアフィ枠プレースホルダ) | Low |
| Button | プライマリCTAは共通して`shadow-[0_4px_0_...]`の疑似立体+`active:translate-y-0.5`で統一。ただしこの値がコンポーネント化されておらず**各ページにコピペ**されている(将来の一括変更が困難) | Medium |
| Spacing | `max-w-md`(5箇所)/`max-w-sm`(1箇所)と概ね統一。個別の`px-4`/`px-5`混在があり、ページ間で外側余白が微妙に異なる(例: selfは`px-5`, mypageは`px-5`だが一部セクションのみ`px-4`) | Low |
| フォント | `font-display`(見出し)/`font-body`(本文)の使い分けは一貫 | 問題なし |
| アイコン | 下部ナビは絵文字ベース、ヘッダーはlucide-reactアイコン。**2系統のアイコン表現が混在**(絵文字 vs SVGアイコンライブラリ) | Low |
| 配色 | ink/gold/torii/paperのトークンに準拠。`/love`ページのみ`rose-*`(Tailwind標準色)を独自追加使用しており、他ページのカラートークン体系から外れている | Medium |
| アニメーション | 演出タイミングの不統一(1章で詳述)。流れ星演出は`MilkyWayBackground`経由で統一 | **Critical(1章と重複)** |
| Modal | `/auction`ページにのみ確認(限定的な実装)。共通Modalコンポーネントは存在しない | Low(現状用途を満たしているため) |
| レスポンシブ崩れ | `max-w-md`基調のモバイルファースト設計。PC表示時は中央寄せで意図的な余白(前回のIA分析で確認済み・CEO了承済みの設計) | 問題なし(設計方針として確認済み) |

## 3. UI品質監査(Apple HIG / Material Design / 現代モバイルUX観点)

現状評価のみ(改善提案は含まない)。

### Apple Human Interface Guidelines 観点

| 評価軸 | 現状 |
|---|---|
| **タップ領域(44×44pt推奨)** | 主要ボタンは`py-3`(12px)〜`py-3.5`(14px)の上下パディング。`text-sm`(14px)フォントと合わせると実測高さは概ね38〜42px程度と推定され、**HIG推奨の44ptにわずかに届いていない可能性**がある(実機計測は未実施) |
| **フォーカス表示** | `:focus-visible`がglobals.cssにグローバル定義されており、キーボード操作時の可視化は担保されている |
| **アクセシビリティラベル** | `Header.tsx`には`aria-label`が複数箇所で付与されているが、**self/love/work等の入力フォームには`aria-label`の付与が確認できず**、スクリーンリーダー対応は限定的 |
| **明確な階層と余白** | セクション単位の`rounded-card`+`border`によるカード分割は一貫しており、視覚的階層は概ね明確 |
| **モーダル/シート** | HIGが重視する「軽量な情報はシート、重要な分岐はダイアログ」という使い分けは、Modal実装が`/auction`にしかないため他画面では判断材料なし |
| **ダークモード** | サービス全体が常時ダーク基調(世界観として意図的)。OSのライト/ダーク切り替えに反応するのはホームのヒーロー画像のみで、UIコンポーネント自体はダークモード固定。**HIGの「システム設定への追従」という観点では部分対応**(ただし本サービスの世界観上、意図的な選択と考えられる) |

### Material Design 観点

| 評価軸 | 現状 |
|---|---|
| **Elevation(影による階層)** | `shadow-lantern`という独自の光彩シャドウのみで、Material的な多段階Elevationシステムは採用されていない(和風の世界観に合わせた独自表現であり、Material準拠を目指した設計ではない) |
| **Ripple/フィードバック** | タップ時の視覚フィードバックは`active:translate-y-0.5`(押し込み表現)のみ。Material標準のRippleエフェクトは未実装 |
| **FAB(Floating Action Button)** | `FloatingCTA`という類似コンポーネントが`ui-common.tsx`に存在し、固定CTAとして機能している |

### 現代的モバイルUX観点

| 評価軸 | 現状 |
|---|---|
| **スケルトンスクリーン/プログレッシブ表示** | `/report`は`ScrollProgress`コンポーネントあり。ローディング中はスピナー中心で、コンテンツ形状を予告するスケルトンUIは未採用 |
| **Pull-to-refresh** | 未実装(ブラウザデフォルト依存) |
| **Optimistic UI** | チャット送信やフォーム送信で楽観的更新は確認されず、すべてAPI応答後に画面更新する同期的な設計 |
| **オフライン耐性** | Service Worker等は未導入。オフライン時の挙動は未定義 |
| **段階的開示(Progressive Disclosure)** | `GlassMosaic`によるモザイク→CTAの「寸止め」パターンが診断系ページ全体で一貫しており、これは現代的なフリーミアムUXとして評価できる設計 |

### 総評

世界観の統一(天の川・和風トークン)という独自軸では一貫性が高い一方、**タップ領域のHIG基準への適合**と**フォーム要素のアクセシビリティラベル**は実機確認・改善の余地がある。Material的なElevation/Rippleは意図的に不採用と見られ、これは和風の世界観との整合を優先した設計判断と解釈できる。
