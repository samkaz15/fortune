## 1. 占い結果表示までの遅延仕様の調査(最優先項目)

### 結論を先に

**バグではない。私(Claude)が2026-07-07の「速度改善」対応時に、正式なCVR仕様(3秒ドラマチックローディング)を誤って縮めてしまったのが直接原因。加えて、self/love/workの3ページは、そもそも正式仕様どおりに実装されたことが一度もなかった(実装漏れ)。** 2つの問題が重なっている。

### 元の設計思想(原典: `docs/design/06_conversion/conversion_spec.md` §5)

2026-07-05付で以下が正式仕様として明文化されている(引用ではなく要約):

| 時刻 | 演出内容 |
|---|---|
| 0.0s | ボタン押下。裏でDB処理/AI生成を開始 |
| 0.5s | 暗転→「糸をたどっています、、」 |
| 1.5s | 「今日の流れと重ねています(65%)」 |
| 2.5s | 「見えました。」+光のエフェクト |
| 3.0s | 結果画面がフェードイン(核心はモザイク状態) |

目的は明記されている:「**バックエンド遅延の隠蔽＋目標勾配効果による期待感の最大化**」。つまり単なる待機時間ではなく、**意図的なCVR施策**として設計されたものだった。「事実でない演出文言は使わない」という制約も付いている。

### 現状の実装との差分

| ページ | 現状 | 仕様との差分 |
|---|---|---|
| `/report`(今日の運勢) | `DramaticLoading`コンポーネントあり。1.2秒・2.4秒でテキスト2段階切替、合計待機時間は明示タイマーなし(API応答待ち) | タイミングが仕様(0.5/1.5/2.5/3.0s)と異なる。暗転・光エフェクトが未実装。**CV1-3実装時(コミット2663b3c、7/6)から簡略版だった**——今回の速度改善の被害ではない |
| `/self` `/love` `/work` | スピナー+1行テキストのみ、**600ms**で結果表示に遷移 | 3段階演出が**最初から実装されていない**。2026-07-06時点では1800msの単純ローディングがあり、まだ「待機時間」としての体裁はあったが、2026-07-07に「速度改善」の名目で600msへ短縮した際、演出としての意味がほぼ失われた |

### 原因の切り分け(ご質問の4分類に対する回答)

- **コード上から消えている**: self/love/workについては、消えたのではなく**最初から実装されていなかった**(3段階演出は一度も存在しない)
- **条件分岐で無効化**: 該当なし
- **別実装に置き換わった**: `/report`は「簡略版の別実装」に置き換わっている(2段階・暗転/光エフェクトなし)。self/love/workは「単純スピナー」という別実装がずっと使われていた
- **バグ**: 狭義のバグではない。**2026-07-07の私の判断ミス**——「動画表示までの速度改善」「UI全体のレスポンス改善」というご指示を受けた際、`conversion_spec.md`のCVR施策としての意図を確認せずに、self/love/workの1800ms待機を「無駄な遅延」と誤認して600msに短縮した。設計意図の見落としによる誤修正。

### 再実装の設計案(実装はまだ行わない)

1. **`DramaticLoading`を共通コンポーネント化**し、`ui-common.tsx`に集約。現状`/report`にのみ存在する実装を、self/love/workからも呼べる形にする
2. **タイミングを仕様書どおり厳密に実装**: 0.5s/1.5s/2.5s/3.0sの4段階、暗転演出、2.5s時点の光エフェクトを追加
3. **文言はページごとに差し替え可能な設計**にする(「事実でない演出文言は使わない」制約を守るため、self=「名前と生年月日を見ています」、love=「ふたりの糸をたどっています」、work=「巡りを整理しています」等、そのページが実際に参照するデータに即した文言をprops等で渡す)
4. **API応答とタイマーの両方を待つ設計**にする: APIが3秒より速く返っても演出は3秒間維持し、逆に3秒より遅い場合は最後の文言(「見えました。」)を保持しながら実際の応答を待つ(Promise.raceではなく両方の完了を待つロジック)
5. 影響範囲: `src/app/self/page.tsx`, `src/app/love/page.tsx`, `src/app/work/page.tsx`, `src/app/report/page.tsx`(タイミング修正), `src/components/ui-common.tsx`(新規共通コンポーネント追加)

## 総合監査サマリー(5領域)

詳細は関連ドキュメント2本に分割。ここでは結論のみ集約する。

### Architecture Audit
`docs/architecture.md`(前回作成)の内容は現行コードと一致していることを本監査でも再確認した。循環依存なし(`fortune-engine`→`decision-report`の一方向)。新たな逸脱は発見されなかった。

### UI Audit / UI品質監査
→ `docs/audit/02_ui_quality_audit.md` 参照。主要発見: `/love`ページのみアフィ枠が共通コンポーネント(`AffSlot`)化されておらずハードコードのまま(3箇所)。それ以外のレイアウト崩れ・レスポンシブ崩れの明確な証拠は静的解析の範囲では未検出(実機スクショでの確認を推奨)。

### Performance Audit / Code Quality Audit
→ `docs/audit/03_code_performance_audit.md` 参照。主要発見: **未使用コード3件**(`ChatWindow.tsx`291行が完全未使用、`itomachi_talk.png`415KBが完全未使用、旧世代API`/api/fortune/result/[id]`が呼び出し元なし)。**メモ化がほぼ皆無**(26ページ中2ページのみ`useMemo`使用、`React.memo`は0件)。「ボタン反応が重い」の主要因はここにある可能性が高い。

### Technical Debt Audit
優先順位付き一覧は`03_code_performance_audit.md`の末尾に集約。Critical 1件(演出ローディングの仕様不一致)、High 3件(未使用コード、メモ化不足、love頁のAffSlot不統一)。

| 原則 | 判定 | 根拠 |
|---|---|---|
