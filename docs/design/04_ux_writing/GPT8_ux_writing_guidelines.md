## Executive Summary

fortune の差別化軸の核心は **「CEO（占術師）の一次情報」** です。

神社レビューと運営者コメントは、
- fortune が単なる「AIアプリ」ではなく「信頼できる占術師のいるサービス」であることを証明する
- ユーザーとブランドの感情的な結びつきを生む
- 他社が絶対に模倣できないコンテンツ（CEO固有の体験・知識）

本ドキュメントでは、その **「声」が正しく伝わるUXライティング方針** を定義します。

---

## 1. 2種類のライティングの定義

| 種類 | 概要 | 主な掲載場所 |
|---|---|---|
| **神社レビュー** | CEOが実際に参詣した神社の体験・占術的見解 | 神社詳細ページ / SNS |
| **運営者コメント** | 占術的根拠・ユーザーへの声がけ・補足説明 | 占い結果ページ / AIチャット補足 / お知らせ |

## 2. 神社レビュー UXライティング方針

### 2-1. 設計思想

```
神社レビューは「旅行口コミ」ではない。

「なぜ、この神社があなたの運気に効くのか」を
占術師として根拠をもって語る。
→ 信頼性（Pillar 1）× 実世界接続（Pillar 3）の体現
```

### 2-2. 必須構成要素（5ブロック）

```
【ブロック1】参詣日・時間帯・天候
  → 「いつ行ったか」の一次情報が信頼感を生む
  例: 「2026年5月の新月の朝、霧雨の中で参詣しました」

【ブロック2】神社の占術的背景
  → 四柱推命・算命学から見たご祭神・場のエネルギー
  例:「月読命は水の性質を持ち、直感・縁・流れを司ります。
       水の五行が弱い方に特に響く場所です。」

【ブロック3】体験・空気感のリアルな描写
  → 五感の描写で「行きたい」を喚起
  例:「境内に一歩入ると、都市の音が遠くなります。
       砂利の音、線香の香り。雑念が自然と抜けていく感覚。」

【ブロック4】おすすめしたいユーザー像（占術タイプ別）
  → パーソナライズ訴求 → App内推奨機能との連携
  例:「四柱推命で"壬（みずのえ）"が弱い方、
       大きな転機を前にいる方には特に足を運んでほしい神社です。」

【ブロック5】行動の誘い（CTA）
  → 「来て、感じて、報告して」のクローズ
  例:「ぜひ参詣して、ツクヨミに感想を話しかけてみてください。
       あなたが感じたものを、一緒に占いに活かしましょう。」
```

### 2-3. 文体原則

| 項目 | 方針 |
|---|---|
| **一人称** | 「わたし」（ツクヨミと統一感） |
| **敬体/常体** | 敬体（です・ます）基本 |
| **文長** | 1文 40字以内を目安 |
| **専門用語** | 使うが必ずすぐに噛み砕く |
| **断定** | 占術的見解は「〜と感じます」「〜でしょう」と余白を残す |
| **感情表現** | 具体的に（「荘厳」より「息が止まるような静けさ」） |

### 2-4. 文字数ガイドライン

| 媒体 | 推奨文字数 | 理由 |
|---|---|---|
| App内 神社詳細ページ | 300〜500字 | スクロールせず読み切れる量 |
| SNS（TikTok/IG Caption） | 100〜200字 | 動画の補足として機能する量 |
| YouTube概要欄 | 500〜800字 | SEOと詳細情報のバランス |
| ブログ/Webコンテンツ | 1,500〜3,000字 | SEO効果 + 深い体験描写 |

### 2-5. NG表現

```
❌「パワースポットとして有名な〇〇神社」
   → 一般観光紹介と区別がつかない

❌「行けば必ず運気が上がります」
   → 断定的。GPT3禁止事項に抵触

❌「〇〇様（ご祭神）は〇〇のご利益があります」
   → 神社公式パンフと同じ内容は差別化にならない

✅「この神社が持つ"水の気"は、わたしの四柱推命分析では
   変化の時期にいる方の流れを整える働きがあると感じます」
```

### 2-6. 神社レビュー テンプレート

```markdown
## 〇〇神社（所在地）

**参詣日**: 2026年〇月〇日（〇曜）/ 〇時頃
**天候**: ○○ / **アクセス**: 最寄り駅から徒歩○分

---

### 占術師として感じた、この神社の"気"

[ブロック1: 参詣時の状況・最初の印象]

[ブロック2: 占術的背景・ご祭神との関係]

### この場所の空気

[ブロック3: 五感の描写・雰囲気]

### こんな方に訪れてほしい

[ブロック4: おすすめユーザー像（占術タイプ別）]

### 参詣後に、ツクヨミに話しかけてみて

[ブロック5: 行動の誘い]

---
⛩ [神社名] ｜ 📍 [住所] ｜ 🗓 [おすすめ参詣時期]
```

## 3. 運営者コメント UXライティング方針

### 3-1. 運営者コメントとは

```
AI（ツクヨミ）の応答とは別に、
「fortune の占術師・運営者」としてのCEOの声を届ける場所。

主な使用箇所:
  - 占い結果ページの「監修者より」
  - 新機能リリース時のお知らせ
  - 重要な占術的節目（年始・二十四節気等）の特別メッセージ
  - App内の「ツクヨミについて」ページ
```

### 3-2. 必須構成要素（3ブロック）

```
【ブロック1】占術師としての根拠の提示
  → 「なぜこの分析が信頼できるか」を一言で
  例:「四柱推命を20年学んできたわたしが、
       このアプリのすべての占術ロジックを監修しています。」

【ブロック2】ユーザーへの共感・寄り添い
  → ツクヨミと近いトーンで、でも「人間」としての温度感
  例:「迷っている夜、不安な朝、このアプリがあなたの
       小さな安心になれたら、それがわたしの願いです。」

【ブロック3】信頼の約束
  → fortune が守り続けるもの
  例:「当たり障りのない占いは作りません。
       あなたの人生の岐路に、本当に役立つ知恵を届けます。」
```

### 3-3. 文体原則

| 項目 | 方針 |
|---|---|
| **一人称** | 「わたし」 |
| **敬体/常体** | 敬体（です・ます）|
| **トーン** | ツクヨミより少し"人間らしく"。温かく、でも権威ある |
| **署名** | 文末に「fortune 占術監修 [CEO名]」 |
| **長さ** | 100〜300字（コンパクトに、深く） |

### 3-4. 使用場面別サンプル

#### ① 占い結果ページ「監修者より」

```
占い結果の下部に常設する短いコメント。

---
【例: 高運気日】
「今日の星の並びは、あなたに味方しています。
 わたしの四柱推命分析でも、この時期に動いた人が
 半年後に大きく変わるケースを何度も見てきました。
 どうか、その一歩を。」

                    fortune 占術監修 ○○

---
【例: 低運気日】
「今日は、意識して"守り"に入ってください。
 何かがうまくいかない日こそ、振り返りが財産になります。
 焦らなくていい。あなたのペースで。」

                    fortune 占術監修 ○○
```

#### ② 新機能リリースお知らせ

```
【例: 相性診断 リリース】

「お待たせしました。
 四柱推命・算命学・姓名判断の3軸で診る
 本格相性診断をリリースします。

 "好きだから合う"ではなく、
 "なぜ合うのか、どこで補い合えるか"まで分かる設計にしました。
 大切な人との関係を、もっと深く理解する道具として
 使っていただけたら嬉しいです。」

                    fortune 占術監修 ○○
```

#### ③ 年始・節目の特別メッセージ

```
【例: 元日メッセージ】

「新年、おめでとうございます。

 今年のあなたの大運を、わたしなりにすべて読みました。
 厳しい星回りの方も、追い風の方も——
 どんな年でも、"知っていること"が最大の守りになります。

 今年もそばにいます。一緒に、いい年にしましょう。」

                    fortune 占術監修 ○○
```

## 4. ツクヨミ（AI）との使い分け

```
【ツクヨミ（AI）】
役割: 24時間対応の「友人」的存在
特徴: 共感・寄り添い・個別対応が強み
場所: チャット・毎日の運勢応答・プッシュ通知

【運営者コメント（CEO）】
役割: 「信頼できる占術師」としての権威
特徴: 一次情報・実績・裏付けが強み
場所: 結果ページ補足・お知らせ・神社レビュー

→ 2つの"声"が共存することで、
  AI の利便性 × 人間の信頼性 を同時に実現
```

## 5. コンテンツ品質チェックリスト

神社レビュー・運営者コメント 公開前チェック。

```
【必須確認項目】
□ 断定的な不幸予言・医療・投資助言が含まれていないか（GPT3禁止事項）
□ 一次情報（実際の体験・知見）が含まれているか
□ 競合コピーと区別がつく独自表現か
□ 文字数がガイドラインの範囲内か
□ 締めにポジティブな余韻が残るか
□ 「fortune 占術監修 〇〇」の署名があるか（運営者コメントのみ）
□ 神社レビューの場合: 5ブロック構成が揃っているか
```

## 6. Claude（実装）への申し送り

```
【App実装要件】

神社レビュー:
  - 神社詳細ページに「監修者レビュー」セクション設置
  - CMS管理（運営者が更新できる構造に）
  - SEO: JSON-LD（Review schema）設定

運営者コメント:
  - 占い結果ページ下部に固定表示エリア確保
  - 運気スコアに応じて「高/低運気」コメントを切り替える設計
  - お知らせページのリッチテキスト対応
  - プッシュ通知との連携（節目メッセージの一斉配信）
```
